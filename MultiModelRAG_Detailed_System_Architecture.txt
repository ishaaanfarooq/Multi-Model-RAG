Chapter 1: EXECUTIVE SUMMARY AND PROJECT VISION
----------------------------------------------------------------------------------------------------

The MultiModelRAG system represents a state-of-the-art approach to Enterprise Knowledge Management, 
specifically designed to solve the persistent challenges of data silos, factual hallucinations, and 
the lack of real-time grounding in standard Large Language Models (LLMs). At its core, the project 
is an autonomous, agentic system that does not simply "search and summarize" but instead "reasons 
and verifies."

In the modern enterprise landscape, critical information is often scattered across diverse formats—
structured databases, unstructured PDF reports, and dynamic web content. Traditional RAG systems 
often fail because they treat every query with a one-size-fits-all retrieval strategy. MultiModelRAG 
diverges from this by implementing an Intelligent Agentic Router that classifies user intent 
before any retrieval takes place. 

The system's primary mission is to provide a "Glass Box" AI experience. Unlike "Black Box" systems 
where the user sees a query and a response with no intermediate visibility, MultiModelRAG utilizes 
Server-Sent Events (SSE) to stream a live "Pipeline Execution Path" to the frontend. This allows 
users to see every step of the agent's thought process—from query rewriting and tool selection 
to cross-encoding, reranking, and the final hallucination check.

By combining local LLM inference (via Ollama) with high-performance vector search (FAISS) and 
live web scraping, MultiModelRAG ensures that responses are not only contextually relevant but 
also historically grounded and real-time accurate.

Chapter 2: THE TECHNOLOGY STACK ECOSYSTEM
----------------------------------------------------------------------------------------------------

The project leverages a modern, high-performance stack optimized for both local development and 
cloud scalability. The choices in technologies were driven by the need for speed, modularity, 
and the ability to run heavy AI workloads (like Llama 3.2) on consumer-grade hardware.

2.1 Backend Framework: FastAPI (Python 3.10+)
FastAPI was chosen for its asynchronous capabilities, which are essential for handling long-running 
AI inference tasks and streaming SSE updates without blocking the event loop. Its native support 
for Pydantic ensures strict data validation across the system.

2.2 AI Orchestration: LangChain
LangChain provides the "glue" that connects the various models. It is used for prompt management, 
document splitting, and managing the interaction between the LLM and the Vector Database.

2.3 Vector Database: FAISS (Facebook AI Similarity Search)
FAISS-CPU is utilized for its incredible speed in high-dimensional vector similarity searches. 
It stores the document embeddings locally, allowing for sub-millisecond retrieval of relevant 
knowledge chunks without the latency or cost of cloud-based vector providers.

2.4 Local LLM Hosting: Ollama
Ollama is the backbone of the system's reasoning. By running Llama 3.2 locally, the system 
ensures data privacy (no data leaves the local machine during inference) and eliminates 
API costs. Llama 3.2 was specifically selected for its balance between reasoning capability 
and speed on 4GB-8GB VRAM systems.

2.5 Frontend Framework: Next.js 15 & React 18
The frontend is built using the latest Next.js features, including the App Router and Server 
Components. It provides a highly responsive, single-page application (SPA) experience that 
renders complex pipeline visualizations and real-time chat updates seamlessly.

2.6 Styling and UI: Tailwind CSS & Lucide Icons
Tailwind CSS provides a utility-first approach to styling, allowing for a premium, custom UI 
that includes dark mode support, glassmorphism effects, and responsive layouts. Lucide provides 
the iconography for the pipeline status steps.

2.7 Data Processing & Visualization: Pandas, Matplotlib, Seaborn
These libraries are used by the "Visualizer Agent" to dynamically generate charts and graphs 
when numerical data is detected in the RAG output.

Chapter 3: BACKEND MODULE BREAKDOWN (FILE-BY-FILE)
----------------------------------------------------------------------------------------------------

The backend is organized into specialized directories, each representing a distinct layer of the 
RAG architecture.

3.1 Root: main.py
This is the entry point of the FastAPI application. It initializes the app, configures CORS 
(Cross-Origin Resource Sharing) to allow communication from the Next.js frontend, and includes 
the API routers. It also manages the static file serving for the dynamically generated charts 
in the `uploads/` directory.

3.2 Orchestrator: master_llm.py
The most critical file in the backend. This "Master Orchestrator" class manages the lifecycle 
of a single query. It handles:
- Query Rewriting: Using the LLM to contextualize conversational history.
- Tool Selection: Interacting with the Agent Router.
- Pipeline Streaming: Yielding SSE events at every stage (Step-by-step progress).
- Hallucination Check: Calling the Verification Module before returning the final answer.
- Memory: Saving the interaction to the `analytical_notebook.md`.

3.3 API Layer: routes.py
Defines the REST and SSE endpoints.
- `/api/chat`: The main streaming endpoint that returns a `EventSourceResponse`.
- `/api/ingest-pdf`: Handles file uploads, parsing, and indexing.
- `/api/crawl`: Takes a URL and triggers the deep web crawler.

3.4 Models Layer:
- agentic_router.py: Uses a specialized prompt to classify queries into "Direct_Chat", 
  "Web_Search", or "Search_Knowledge_Base".
- generation.py: Contains the sophisticated "Senior Research Analyst" prompts. It supports 
  multiple modes (Conversational vs. Analytical) to ensure greetings are short and reports 
  are thorough.
- embedding.py: Manages the `SentenceTransformers` model (All-MiniLM-L6-V2) used to convert 
  text into 384-dimensional vectors.

3.5 Retrieval Layer:
- vector_db.py: Handles the creation, loading, and searching of the FAISS index. It includes 
  logic for persistence and metadata management.
- web_search.py: A multi-tier search engine that scrapes DuckDuckGo, falls back to the 
  Wikipedia REST API, and uses Google News RSS as a third-tier safety net.
- reranker.py: Implements a Cross-Encoder model (BGE-Reranker) to take the top 10 results 
  from FAISS and re-sort them based on semantic relevance to the query, reducing the context 
  noise for the LLM.
- visualizer.py: A specialized agent that writes and executes temporary Python scripts 
  using Matplotlib to create charts based on numerical data in the answer.

3.6 Ingestion Layer:
- document_parser.py: Uses `PyPDF` and `LangChain` text splitters to break large documents 
  into manageable, overlapping chunks (approx. 1000 characters).
- web_crawler.py: A recursive scraper that follows links within a domain to build a 
  comprehensive knowledge base from a single website.

3.7 Verification Layer: verifier.py
The "Fact-Checker." It takes the generated answer and the raw source context and asks the 
LLM to identify any "Significant Fabricated Hallucinations." If found, it triggers a 
"FAIL" status and adds a warning to the UI.

Chapter 4: FRONTEND ARCHITECTURE AND USER EXPERIENCE
----------------------------------------------------------------------------------------------------

The frontend is designed to be as informative as it is functional.

4.1 The Core Page: page.tsx
Manages the global state of the application, including the message history, the currently

4.2 The Pipeline Visualizer (Timeline Component)
A side panel that renders the "Pipeline Execution Path." Each step (e.g., "Web Search",

sub-text (e.g., "Retrieved 5 live web results").

4.3 The Chat Interface
A premium, centered chat layout with smooth animations. It supports:
- Real-time Markdown rendering.
- Code syntax highlighting.
- Embedded chart rendering (for the Visualizer Agent output).
- Source attribution (showing exactly which URLs or documents provided the info).

4.4 Global Styles: globals.css
Custom Tailwind configuration that implements a "Soft Wheat" and "Zinc/Amber" color 
palette, giving the application a professional, academic, yet modern feel.

Chapter 5: THE RAG LIFECYCLE (STEP-BY-STEP)
----------------------------------------------------------------------------------------------------

When a user types "Compare Nvidia and Google revenues in 2025," the following sequence occurs:

1.  Input Reception: Next.js captures the query and sends an SSE request to `/api/chat`.
2.  Query Rewriting: If there is a history, the Master Orchestrator rewrites the query for clarity.
3.  Agent Routing: The Agent Router analyzes the query and selects "Web_Search."
4.  Web Search: The system scrapes DuckDuckGo and Wikipedia, retrieving 5 high-quality extracts.
5.  Verification Ingress: The source URLs are captured for the "Sources" section of the UI.
6.  Reranking: The BGE-Reranker ensures the most data-rich extracts are at the top.
7.  Synthesis: The LLM (Llama 3.2) writes a 3-paragraph analytical report based ONLY on the extracts.
8.  Verification: The Verifier module checks for hallucinations. If it sees $40B instead of $100B, 
    it flags a FAIL.
9.  Visualization: The Visualizer Agent detects the two company names and their revenue figures, 
    generates a Bar Chart, and saves it.
10. Final Delivery: The frontend receives the final answer, the sources, the chart, and the 
    verification status, rendering them all in a single cohesive response block.

Chapter 6: DEPLOYMENT, SECURITY, AND SCALABILITY
----------------------------------------------------------------------------------------------------

6.1 Dockerization
The project includes a `docker-compose.yml` that orchestrates the backend and frontend 
containers. This ensures a consistent environment across different developer machines and 
production servers.

6.2 Local Security
Because the system runs Ollama and FAISS locally, sensitive enterprise data never touches 
external LLM APIs. This is a critical feature for government or legal sectors where data 
residency is a requirement.

6.3 Future Scalability
The architecture is designed to swap the local FAISS index for a distributed vector store 
(like Pinecone or Milvus) and the local Ollama instance for an enterprise LLM (like 
Vertex AI or OpenAI) with minimal code changes in the `models/` and `retrieval/` layers.

Chapter 8: API SPECIFICATION AND DATA SCHEMAS
----------------------------------------------------------------------------------------------------

The MultiModelRAG API is built on a robust, type-safe architecture using FastAPI and Pydantic. 
Below are the detailed schemas for the primary communication channels.

8.1 The Chat Stream Endpoint (SSE)
Path: `GET /api/chat`
Query Parameters:
- `query` (string): The user's input question.
- `history` (optional string): A JSON-stringified array of previous messages for context.

Response Format (Server-Sent Events):
Each event is a JSON object with the following structure:
```json
{
  "model": "Name of the active agent/module",
  "status": "Processing | Completed | Error",
  "action": "Description of the current task",
  "details": {
    "answer": "The generated text (only in the final step)",
    "sources": ["List of URLs/filenames"],
    "chart": "URL to the generated image",
    "warning": "Optional hallucination alert"
  }
}
```

8.2 Document Ingestion Endpoint
Path: `POST /api/ingest-pdf`
Content-Type: `multipart/form-data`
Payload:
- `file`: The PDF binary.

Processing Workflow:
1.  The file is saved to the `backend/uploads/` directory with a unique UUID prefix.
2.  `PyPDFLoader` extracts raw text from the pages.
3.  `RecursiveCharacterTextSplitter` chunks the text with a 10% overlap to maintain context.
4.  The `LocalEmbeddingModel` generates 384-dimensional vectors for each chunk.
5.  Vectors are added to the FAISS index and the index is persisted to disk.

8.3 Web Crawling Endpoint
Path: `POST /api/crawl`
Payload: `{"url": "https://example.com", "depth": 2}`

Chapter 9: PROMPT ENGINEERING AND AGENTIC INSTRUCTION SETS
----------------------------------------------------------------------------------------------------

The reliability of the system stems from highly specific "System Prompts" that constrain 
the behavior of the local Llama 3.2 model.

9.1 The Agentic Router Prompt
The router is the system's "switchboard." It uses a zero-shot classification prompt 
to decide between search tools. 
Constraint: "You MUST output ONLY the tool name. Do not explain your reasoning."
Logic: It prioritizes "Web_Search" for any query containing proper nouns (Nvidia, Tesla) 
or temporal indicators (today, 2025).

9.2 The Senior Research Analyst (Generation) Prompt
This prompt uses the "Chain of Thought" principle implicitly.
Rules:
- "Synthesize the provided context into a high-level, data-driven report."
- "NO LISTS: Use professional prose only."
- "If data is missing, admit it. Do not hallucinate."
This ensures the final output feels like a professional memo rather than a generic AI chat.

9.3 The Verification Module Prompt
This is a "self-correction" mechanism.
Query: "Compare the Answer against the Context."
Output: "PASS" or "FAIL".
The LLM is asked to look specifically for "Numerical Hallucinations"—cases where the 
answer cites a number not present in the retrieval context.

Chapter 10: FRONTEND COMPONENT INTERNALS
----------------------------------------------------------------------------------------------------

The frontend is built for low-latency updates and high visibility.

10.1 The Timeline Component (Pipeline Visualizer)
This component maintains a reactive state of "Steps." As SSE events arrive, the 
Timeline component updates the status icon (Spinner -> Checkmark) and adds the 
detailed sub-text to the step description. 
Tech: Framer Motion for smooth transitions between steps.

10.2 The Data Visualizer Integration
When the `Visualizer Agent` succeeds, the backend provides a URL (e.g., `/uploads/chart_xyz.png`).
The frontend detects this in the message metadata and renders a specialized `<ChartDisplay />` 
component within the chat bubble, allowing for a seamless blend of text and graphics.

10.3 Source Attribution Tooltips
Each source URL is processed to show a favicon and a truncated domain name. 
Hovering over a source reveals the specific text snippet that was used to 
generate that portion of the answer, providing full auditability.

Chapter 11: FAISS INDEXING AND SEMANTIC SEARCH DEPTH
----------------------------------------------------------------------------------------------------

11.1 Embedding Generation
We utilize the `sentence-transformers/all-MiniLM-L6-v2` model. This model is 
small enough (approx. 80MB) to run efficiently in the CPU-bound backend while 
maintaining high semantic accuracy.

11.2 Similarity Metrics
The system uses L2 (Euclidean) distance for similarity search. While Inner Product (IP) 
is common, L2 provides more stable results for the normalized embeddings generated 
by the MiniLM model.

11.3 Top-K and Reranking Strategy
1.  Initial Retrieval: FAISS fetches the Top-10 candidates.
2.  Reranking: The `BGE-Reranker-v2-m3` model scores these 10 against the query.
3.  Final Selection: Only the Top-5 highest-scoring chunks are passed to the 
    LLM's context window. This "Double-Filter" approach significantly reduces 
    the noise and prevents "The Lost in the Middle" phenomenon in LLMs.

Chapter 15: THE ANALYTICAL NOTEBOOK (PERSISTENCE LAYER)
----------------------------------------------------------------------------------------------------

The `analytical_notebook.md` is a unique feature of MultiModelRAG that provides a 
long-term "research memory."

15.1 Structure and Purpose
Unlike a database which stores data in structured tables, the Notebook stores 
results in a human-readable Markdown format. This allows researchers to 
open the file in any text editor and see a historical timeline of queries, 
generated answers, and the specific sources that supported them.

15.2 Automated Logging Logic
Every time the `MasterOrchestrator` completes a pipeline run, it invokes the 
`NotebookMemory` class. This class appends a new entry with:
- Timestamp (ISO 8601)
- The User Query (Original and Rewritten)
- The synthesized Final Answer
- A list of all source URLs and local filenames used.

15.3 Research Utilization
By maintaining this notebook, the system allows for "Retrospective Auditing." 
If a user questions an answer weeks later, the exact source context used 
at that moment is preserved in the notebook for verification.

Chapter 16: SERVER-SENT EVENTS (SSE) IMPLEMENTATION DEPTH
----------------------------------------------------------------------------------------------------

The real-time streaming is managed via the `sse-starlette` library in the backend.

16.1 Event Lifecycle Mapping
- `START`: Initial connection established.
- `ROUTING`: Agent router is processing the intent.
- `RETRIEVING`: Vector DB or Web Search is active.
- `GENERATING`: LLM is synthesizing the report.
- `VERIFYING`: Hallucination detection is running.
- `FINISH`: Final payload with answer and chart URL sent.

16.2 Frontend Consumption (React Hooks)
The `page.tsx` uses a custom `useSSE` hook to manage the connection. 
When an event arrives:
1. The JSON is parsed.
2. The `currentStep` state is updated.
3. If the event contains a `details.answer` snippet, it is appended 
   to the active message object in the UI.

Chapter 17: ERROR HANDLING AND AGENTIC RECOVERY
----------------------------------------------------------------------------------------------------

A primary strength of an agentic system is its ability to handle failures 
at individual steps without crashing the entire session.

17.1 Web Search Fallbacks
If the DuckDuckGo scraper returns a 403 Forbidden error (due to rate limiting), 
the system doesn't stop. It automatically "falls back" to the Wikipedia REST API. 
If that also fails, it tries the Google News RSS feed. This ensures the 
generation layer almost always has some context to work with.

17.2 Local LLM Timeouts
Ollama can sometimes be slow on low-VRAM systems. The backend includes a 
60-second timeout for LLM inference. If a timeout occurs, the orchestrator 
emits an "Error" status to the frontend and provides a graceful fallback 
message to the user, suggesting they try a simpler query.

17.3 Ingestion Validation
During PDF upload, the system checks for password protection and 
corrupted streams. If a file is unreadable, the `document_parser.py` 
catches the exception and notifies the user through the API response, 
preventing broken vectors from entering the FAISS index.
